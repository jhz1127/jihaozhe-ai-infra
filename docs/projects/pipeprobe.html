PipeProbe · DataLoader 性能诊断

在不改写 PyTorch DataLoader 原生传输方式的前提下，定位数据链路中的长尾与等待来源。

京东实习项目 · Python / PyTorch / 可观测性

问题与方案

DataLoader 的慢可能来自读取、解析、分词、组批、worker 排队或主进程取数；仅看平均耗时无法分辨根因。PipeProbe 为每个 batch 建立严格对齐的时间线，并将诊断信息随原始 batch 一起返回，因此不会额外引入 IPC 队列或后台聚合线程。

时序架构

sequenceDiagram
  participant W as Worker 进程
  participant DL as DataLoader 原生通道
  participant M as Main Process
  W->>W: span(read)：Parquet 读取
  W->>W: span(parse / tokenize / collate)
  W->>W: 封装 TimedBatch(payload, t_ready, summary)
  W->>DL: 随原生 batch 通道传回
  Note over DL: 同一个 mp.Queue
无额外 IPC
  DL->>M: 解包 TimedBatch
  M->>M: t_fetch - t_ready = queue dwell
  M->>M: 当前时间 - t_fetch = fetch wait
  M->>M: 聚合 per-worker P50 / P90 / P99

诊断数据与 payload 同路返回，因而能够精确对应到具体 batch、worker 和 rank。

关键设计

  • 支持嵌套 span 与 exclusive time，区分一个阶段自身耗时和其子阶段耗时。
  • 同时观察 worker wall time、队列驻留、主进程 fetch wait 与训练节奏，避免只用单一指标判断瓶颈。
  • 按 worker、rank 维度报告分位数：全局 P99 往往只是某个慢 worker 的异常，必须保留归属信息。
  • 封装层保持对训练代码低侵入，可在原有 DataLoader 调用链中逐步启用。

结果

PipeProbe 将“DataLoader 很慢”的模糊现象拆分为可归因的阶段和队列事件，使后续的预取、并发数和数据格式优化都有明确的实验依据。
