分离式 DataLoader

将 CPU 数据准备、控制调度与 GPU 数据搬运解耦，让训练计算与数据供给真正并行。

京东实习项目 · PyTorch / Ray / RDMA / GPUDirect RDMA

背景与目标

传统 DataLoader 由训练进程直接承担读取、解析和组批。当 Parquet 流式读取或复杂预处理出现抖动时，GPU 会等待数据；扩展 worker 又会引入 Python 进程和跨节点传输开销。我将数据生产移出训练主进程，在不改变训练侧消费体验的前提下，构建可横向扩展的 M×N 数据供给架构。

整体架构

  subgraph P["Producer × M · 独立进程或节点"]
    DS["Parquet Streaming"] --> PP["filter / sort / pack<br/>CPU 多核并行"]
    PP --> MR["OT ctx.malloc<br/>预注册 RDMA MR"]
  end
  subgraph Ctrl["控制面 · Ray Queue · KB 级"]
    MR -->|"push FatPtr / batch_id<br/>约 200B 每 batch"| DQ["Dispatcher actor<br/>中央队列 · pull-based<br/>链式反压"]
  end
  subgraph Data["数据面 · MB 级 · 不经过 Ray"]
    MR -.->|"one-sided RDMA READ<br/>GPUDirect RDMA<br/>直写 GPU HBM"| GF["Consumer GPU HBM"]
  end
  subgraph C["Consumer × N · torchrun rank"]
    DQ -->|"pop FatPtr"| SCH["后台线程预取<br/>本地 deque"]
    SCH -->|"copy_to_local"| GF
    GF --> TR["训练主循环 · matmul"]
    TR -.->|"ack batch_id"| MR
  end

Ray 只传递 FatPtr、batch_id 与确认信号；大体积样本由 RDMA 直接进入 GPU 显存。

关键设计

  • 控制面与数据面分离：控制面负责全局调度、容量反馈与反压，数据面绕过 Ray，避免大对象序列化和中转。
  • 拉取式调度：Consumer 仅在本地预取队列有空位时请求下一批，Dispatcher 将背压一路传回 Producer，避免数据堆积。
  • 显存直达：Producer 返回可定位远端内存的 FatPtr；Consumer 通过 one-sided RDMA READ 和 GPUDirect RDMA 将数据写入 HBM。
  • 可靠恢复：在数据生产侧记录文件、行号和 batch 状态，使行级断点续训与分片、组批流程能够协同。

结果

31.1k → 41.1ksamples/s
+32%端到端数据吞吐
M×N生产与训练独立扩缩

该方案把训练主线程从数据准备中释放出来，既提升稳定吞吐，也为跨节点数据供给保留了清晰的扩展边界。
